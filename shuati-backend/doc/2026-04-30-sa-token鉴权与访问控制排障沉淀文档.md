# 2026-04-30 Sa-Token 鉴权与访问控制排障沉淀文档

## 1. 会话背景

本次会话的核心目标不是新增业务功能，而是把项目从“局部能跑”修到“整条登录、鉴权、接口访问链路稳定可用”。

排障范围覆盖了：

1. Sentinel 本地规则加载失败导致 Spring Boot 启动失败
2. Nacos 黑名单监听器与布隆过滤器初始化异常
3. 旧自定义鉴权 `@AuthCheck` 与新 `sa-token` 并存引发的 AOP 启动错误

7. 管理页隐藏弹窗预加载题库列表，误触发 Sentinel 限流提示

本次工作的本质，是把权限体系从“零散修补”整理成“可解释、可验证、可继续扩展”的状态。

## 2. 本次问题总览

本次实际遇到的问题可以归纳为五类：

### 2.1 启动期错误

- `JSONObject cannot be cast to FlowRule`
- `NegativeArraySizeException`
- `Annotation type com.syong.shuati.annotation.AuthCheck does not have runtime retention`

- 首页需要开放访问

## 3. 关键问题与修复

## 3.1 Sentinel 规则加载失败

### 现象

启动时报错：

- `com.alibaba.fastjson.JSONObject cannot be cast to FlowRule`

### 根因

`SentinelRulesManager` 中使用了：

- `com.alibaba.fastjson.JSON.parseObject(...)`
- 但 `TypeReference` 错误引用成了 `cn.hutool.core.lang.TypeReference`

这会导致 Fastjson 没有按 `List<FlowRule>` 正确反序列化，而是得到 `JSONObject` 列表，最终在 Sentinel 内部强转失败。

### 修复

将：

- `cn.hutool.core.lang.TypeReference`

替换为：

- `com.alibaba.fastjson.TypeReference`

涉及文件：

- [SentinelRulesManager.java](/e:/software/solo-website-cursor/shuati-backend/src/main/java/com/syong/shuati/sentinel/SentinelRulesManager.java)

---

## 3.2 Nacos 黑名单监听器问题

### 现象

`NacosListener` 初始实现存在多个问题：

- `@Value` 错误引用为 Lombok 的 `Value`
- 缺失 `Nacos Listener` 的正确 import
- 监听线程池未关闭
- 黑名单工具类存在空指针窗口

### 修复

1. 将 `@Value` 改为 Spring 的 `org.springframework.beans.factory.annotation.Value`
2. 补充 `com.alibaba.nacos.api.config.listener.Listener`
3. 将线程池提升为类字段，并在 `destroy()` 中 `shutdown()`
4. 给 `BlackIpUtils` 补默认布隆过滤器与空值保护

涉及文件：

## 3.3 布隆过滤器容量参数误用

### 现象

启动时报错：

- `NegativeArraySizeException`

### 根因

`Hutool BitMapBloomFilter(int)` 的构造参数语义不是“预计元素个数”，而是“内存大小 MB”。

原实现把类似 `958506` 这样的数字直接传给构造函数，内部折算后导致位图数组过大，在 `IntMap` 中溢出。

### 修复

改成固定、保守、可控的配置：

- 默认过滤器：`8 MB`
- 重建过滤器：`16 MB`

并显式注释说明该参数单位为 `MB`。

涉及文件：

- [BlackIpUtils.java](/e:/software/solo-website-cursor/shuati-backend/src/main/java/com/syong/shuati/blackfilter/BlackIpUtils.java)

## 3.4 旧鉴权链路与 Sa-Token 冲突

### 现象

启动时报错：

- `Annotation type com.syong.shuati.annotation.AuthCheck does not have runtime retention`

### 根因

项目已经切换到 `sa-token`，但旧的：

- `@AuthCheck`
- `AuthInterceptor`

仍被 Spring AOP 扫描。

同时 `AuthCheck` 上缺失运行时注解保留策略，导致切点表达式 `@annotation(authCheck)` 在启动时解析失败。

### 修复决策

既然项目已使用 `sa-token`，不再修补旧链路，而是直接清理：

1. 删除 `AuthInterceptor`
2. 删除 `AuthCheck`
3. 删除控制器中残留但未使用的 `AuthCheck import`
4. 统一使用 `SaCheckRole` 与 `SaInterceptor + StpUtil.checkLogin()`

涉及文件：

- 已删除 [AuthInterceptor.java](/e:/software/solo-website-cursor/shuati-backend/src/main/java/com/syong/shuati/aop/AuthInterceptor.java)
- 已删除 [AuthCheck.java](/e:/software/solo-website-cursor/shuati-backend/src/main/java/com/syong/shuati/annotation/AuthCheck.java)

## 3.8 SSR 请求未携带 Cookie

### 现象

即使浏览器已经登录，题库详情页、题目详情页等 SSR 页面仍然拿不到数据。

### 根因

Next.js 服务端渲染调用后端接口时，没有自动把当前请求里的 cookie 转发给 Spring Boot。

这会导致：

- 浏览器里已登录
- 但服务端发起的接口请求没有 token
- 后端仍判定未登录

### 修复

重写前端请求层：

- [request.ts](/e:/software/solo-website-cursor/mianshiya-next-frontend/src/libs/request.ts)

在服务端环境下通过 `next/headers` 读取当前 cookie，并把 `Cookie` 请求头自动转发到后端。

同时补充浏览器环境判断，避免 SSR 中直接访问 `window`。

## 3.9 OPTIONS 预检请求被 Sa-Token 拦截

### 现象

日志中大量出现：

- `FrameworkServlet.doOptions`
- `NotLoginException: 未能读取到有效 token`

并且很多前端请求最终表现为“系统错误”。

### 根因

浏览器在发送部分 `POST` / `JSON` 请求前会先发 `OPTIONS` 预检。

但后端全局 `StpUtil.checkLogin()` 没有区分预检请求，导致业务请求在真正发送前就先失败。

### 修复

在 `SaTokenConfigure` 中改为：

1. 通过 `SaHolder.getRequest().getMethod()` 获取 HTTP 方法
2. 对 `OPTIONS` 请求直接放行
3. 补充放行 `/error`，避免错误页再被登录拦截形成异常循环

涉及文件：

- [SaTokenConfigure.java](/e:/software/solo-website-cursor/shuati-backend/src/main/java/com/syong/shuati/satoken/SaTokenConfigure.java)

## 6. 关键经验沉淀

## 6.1 鉴权迁移必须“清旧链路”

从自定义 AOP 鉴权切换到 `sa-token` 时，不能只“加新方案”，还必须确认旧注解、旧切面、旧 import 是否完全退出运行链路。

否则很容易出现：

- 启动时 AOP 解析异常
- 两套权限规则互相打架
- 后续排障难以判断问题来源

## 6.2 全局登录拦截不能只考虑业务接口

真实项目里至少要同时考虑：

1. 匿名入口，如登录、注册、接口文档、首页公开数据
2. 浏览器预检请求 `OPTIONS`
3. 错误页 `/error`
4. SSR 与 CSR 两种请求来源

如果只写一条 `StpUtil.checkLogin()` 覆盖 `/**`，几乎一定会在后续接入中不断踩坑。

## 6.3 SSR 场景下“浏览器已登录”不等于“服务端请求已登录”

这是 Next.js 接后端鉴权时最容易忽略的一点：

- 浏览器页面有 cookie
- 不代表服务端发起的接口请求自动带 cookie

所以只要页面是 SSR / Server Component，就要明确设计 cookie 透传。

## 6.4 Sentinel 误报很多时候不是接口本身慢，而是前端重复请求

本次题目管理页的“系统压力过大”就是典型例子：

- 真正有问题的不是表格主查询
- 而是隐藏弹窗在后台重复打同一个题库列表接口

因此限流问题排查不能只盯后端规则，也要看前端是否存在隐式并发请求。

## 7. 后续建议

建议后续继续推进以下优化：

1. 将匿名放行路径统一抽成常量或配置，避免 `SaTokenConfigure` 中字符串散落
2. 为前端受保护页面建立统一的路由白名单 / 黑名单策略，减少多层判断重复
3. 为题库列表、标签列表等低频基础数据增加前端缓存，减少重复请求
4. 如果 Sentinel 继续保留，建议补充更明确的限流监控与日志字段，便于定位是哪个页面、哪个组件、哪个 IP 触发
5. 补一轮端到端联调文档，明确匿名页面、登录页面、管理员页面各自允许访问的接口集

## 8. 最终结论

本次会话的核心成果，不是单点修复某一个报错，而是把“登录、权限、公开访问、接口文档、SSR 请求、预检请求、前端重复请求”这几条原本互相缠绕的链路逐步拆开并重新梳理清楚。

最终形成了以下稳定规则：

1. `sa-token` 成为唯一鉴权方案
2. 首页与首页公开数据允许匿名访问
3. 题库、题目、详情、测试、用户中心和管理页必须登录后访问
4. 浏览器预检请求与错误页不再参与登录判定
5. SSR 请求能够透传 cookie，前后端登录态保持一致
6. 页面级隐藏组件不再无意义预加载数据，避免误伤 Sentinel

这份文档可作为：

1. 本次权限与访问控制排障的完整沉淀
2. 后续继续优化 `sa-token` 接入的背景材料
3. 项目答辩或阶段汇报时关于“系统稳定性修复”的说明依据
